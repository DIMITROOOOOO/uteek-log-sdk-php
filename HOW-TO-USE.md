# How to Use the UTEEK Log Tracker SDK

This guide explains how to integrate the SDK into a real project (using Laravel as the primary example) and what each file in the SDK does.

---

## 1. How It Works: The Big Picture

The SDK's job is to capture errors and logs from your PHP application and send them to your Node.js backend. It does this efficiently by:

1.  **Capturing:** It hooks into PHP's native error/exception handlers and framework-specific logging systems (like Monolog in Laravel/Symfony).
2.  **Enriching:** It automatically adds context to every log: the user's IP, the request URL, the server name, etc.
3.  **Buffering:** It collects logs in memory instead of sending one HTTP request for every single error.
4.  **Flushing:** It sends the buffered logs as a single JSON batch to your Node.js `/ingest` endpoint at the end of the request.

```
Your Laravel App
 │
 ├─ Log::error("DB error") ──► Laravel Logging System (Monolog)
 │                                 │
 │                                 └─ LogTrackerHandler (from our SDK)
 │                                      │
 │                                      └─ LogTrackerClient (core SDK class)
 │                                           │
 └─ try/catch(Exception $e)                   ├─ Buffers the log in memory
    └─ $client->captureException($e) ─────────┘
                                               │
                                               ▼
                                         (end of request)
                                               │
                                               ▼
                                          POST /ingest
                                               │
                                               ▼
                                       Node.js Backend
```

## 2. Real-Time Integration (Laravel Example)

Follow these steps to get real-time, automatic error tracking in your Laravel project.

### Step 1: Install the Package

In your Laravel project directory, run:

```bash
composer require uteek/log-tracker-sdk
```

### Step 2: Publish the Configuration

The SDK comes with a Laravel Service Provider that's auto-discovered. Run the `vendor:publish` command to copy the configuration file to your project's `config` directory.

```bash
php artisan vendor:publish --tag=logtracker-config
```

This creates the file `config/logtracker.php`.

### Step 3: Configure Your `.env` File

Open your `.env` file and add the credentials for the project you created in the UTEEK monitoring dashboard.

```env
LOGTRACKER_ENABLED=true
LOGTRACKER_API_URL="http://your-node-backend.com"
LOGTRACKER_PROJECT_ID="proj_abc123xyz"
LOGTRACKER_API_KEY="ltk_prod_your_secret_api_key"
LOGTRACKER_ENV="${APP_ENV}"
```

### Step 4: Route Laravel's Logs to the SDK

To automatically capture every log generated by `Log::info()`, `Log::error()`, etc., you need to add a new channel to your `config/logging.php` file.

1.  **Add the `logtracker` channel:**

    ```php
    // config/logging.php

    'channels' => [
        // ... other channels like 'single', 'daily'

        'logtracker' => [
            'driver'  => 'monolog',
            'handler' => \Uteek\LogTracker\Laravel\LogTrackerHandler::class,
            'level'   => 'warning', // Only send warnings, errors, and critical logs
        ],
    ],
    ```

2.  **Add the new channel to the `stack`:**

    The `stack` channel lets you send logs to multiple places at once (e.g., to a local file AND to UTEEK).

    ```php
    // config/logging.php

    'channels' => [
        'stack' => [
            'driver' => 'stack',
            'channels' => ['single', 'logtracker'], // <-- Add 'logtracker' here
            'ignore_exceptions' => false,
        ],

        // ...
    ],
    ```

**That's it!** Your Laravel application will now automatically send all `warning`, `error`, and `critical` logs and exceptions to your Node.js backend in real-time.

### Step 5: Manual Logging & User Context (Optional)

Sometimes you want to log something manually or add context about the current user.

You can get the client instance from the service container. For example, in a controller:

```php
use Illuminate\Support\Facades\Log;
use Uteek\LogTracker\LogTrackerClient;

class PaymentController extends Controller
{
    public function __construct(private LogTrackerClient $client)
    {
        // Or use: $this->client = app('logtracker');
    }

    public function processPayment()
    {
        // Add context about the logged-in user to all subsequent logs
        if (auth()->check()) {
            $this->client->setUser([
                'id'    => auth()->id(),
                'email' => auth()->user()->email,
                'name'  => auth()->user()->name,
            ]);
        }

        try {
            // ... payment logic ...
            Log::info('Payment initiated by user.', ['amount' => 100]);

        } catch (\Exception $e) {
            // Capture the exception with extra context
            $this->client->captureException($e, [
                'order_id' => 'ORD-123',
                'payment_gateway' => 'Stripe'
            ]);

            // You can also just use Laravel's logger, it will be captured automatically
            // Log::error($e->getMessage(), ['exception' => $e]);

            return response()->json(['error' => 'Payment failed'], 500);
        }
    }
}
```

## 3. SDK File Explanations

| File | Purpose |
|---|---|
| **`src/LogTrackerClient.php`** | **The Core Engine.** This class does all the heavy lifting. It manages the configuration, validates the API key, buffers logs, enriches them with context (request, server, user), and sends the final JSON payload to the Node.js backend via cURL. All other files ultimately use this class. |
| **`src/Client.php`** | **Backward Compatibility.** This is a simple wrapper that extends `LogTrackerClient`. It was kept from the prototype version so that any old code using `new Client()` doesn't break immediately. It maps the old `ingest_url` config key to the new `api_url`. New projects should use `LogTrackerClient` directly. |
| **`src/Laravel/LogTrackerServiceProvider.php`** | **Laravel Integration.** This is a standard Laravel Service Provider. It tells Laravel how to create the `LogTrackerClient` instance (reading credentials from `config/logtracker.php`) and registers it in the service container, so you can use dependency injection or `app('logtracker')`. It also makes the config file "publishable". |
| **`src/Laravel/LogTrackerHandler.php`** | **The Laravel Bridge.** This is a custom [Monolog Handler](https://github.com/Seldaek/monolog). Monolog is the powerful logging library that Laravel uses under the hood. This handler acts as a bridge: whenever Laravel's `Log` facade is used, Monolog passes the log record to this handler, which then calls the appropriate method (`->info()`, `->error()`, etc.) on our `LogTrackerClient`. This is how automatic logging works. |
| **`src/WordPress/LogTrackerPlugin.php`** | **WordPress Helper.** A convenience class for WordPress environments. It provides a simple `register()` method that sets up the global error/exception handlers and hooks into WordPress-specific actions to automatically capture the logged-in WordPress user's details. |
| **`src/Symfony/LogTrackerHandler.php`** | **The Symfony Bridge.** Similar to the Laravel handler, this is a Monolog handler tailored for Symfony projects. It allows developers to configure it in `monolog.yaml` to route Symfony's logs to our SDK. |
| **`config/logtracker.php`** | **Laravel Config Template.** This is the configuration file that gets copied into a Laravel project's `config` directory. It provides a clean way for developers to set their credentials, pulling values from the `.env` file. |
| **`BACKEND-INTEGRATION.md`** | **Node.js Developer Guide.** This file explains to the backend developer exactly what endpoints they need to build (`/ping`, `/ingest`), how authentication works (`X-API-Key` header), and what the JSON payload structure looks like. |
| **`tests/`** | **Testing Suite.** Contains a mock Node.js server (`mock-server.php`) and a test runner (`run-test.php`) to ensure the SDK works as expected without needing the real backend. |
